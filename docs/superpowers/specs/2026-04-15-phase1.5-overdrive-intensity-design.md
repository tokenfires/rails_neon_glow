# Phase 1.5 — Overdrive Intensity & Per-Theme Defaults Design Spec

- **Date:** 2026-04-15
- **Status:** Draft — ready for implementation
- **Branch:** `feat/phase-1.5-overdrive-intensity`
- **Authors:** Claude (design) + TK / TokenFires (direction)
- **Companion to:** `2026-04-10-hardware-themes-and-typography-design.md` (the v2 hardware-themes spec, which originally noted Overdrive as a Phase 1.5 deliverable)

## Summary

Add a fourth intensity level — **Overdrive** — to `neon_glow`, sitting above `intense`. Establish per-theme default intensities so that selecting any palette automatically applies the intensity level its visual identity calls for, with manual override still available.

This is a small additive feature. It does not modify any existing intensity values or palette colors. It does add a new intensity class (`neon-overdrive`), a new dropdown option in the navbar, an auto-apply mechanism in the Stimulus controller, and a documented set of per-theme intensity defaults.

## Why we're doing this

Two reasons, both surfaced during TK's visual review of Phase 1:

**1. The "overdriven neon" character is missing from the intensity range.** The existing `subtle / medium / intense` levels cover "professional" through "loud," but they don't reach the bloom-past-normal-operating-range register that real overdriven neon tubes exhibit — light expanding beyond the element boundary, halos visible against dark surroundings, the streetlamp-against-night-sky feel. TK's framing was sharp: "what happens when a neon light is overdriven? Its intensity goes up. And at night on dark streets, where there are few or no other light sources, neon's intensity in that extreme low light has a natural expanding overdriven effect." That register is what `Overdrive` is for. The name is also thematically perfect for the project — `neon_glow` is literally about overdriving visual presence.

**2. Some palettes have a "right" intensity that gets lost without per-theme defaults.** Cherenkov reads strongest at Overdrive (the underwater bloom needs the extra spread). Nixie reads strongest at Intense (modern nostalgic reproductions lean intense; historically accurate would be medium). Social (2010s corporate UI) reads strongest at Subtle. Without per-theme defaults, every new visitor sees every palette at the system default (medium), which under-sells the palettes that need more or less to land. Per-theme defaults make the intent of each palette land on first viewing.

## Design

### The Overdrive intensity class

Existing intensity values (from `tokens.css`):

| Field | subtle | medium | intense |
|---|---|---|---|
| `--ng-glow-spread` | 4px | 8px | 12px |
| `--ng-glow-blur` | 8px | 16px | 28px |
| `--ng-glow-opacity` | 0.3 | 0.5 | 0.7 |
| `--ng-glow-text-blur` | 4px | 8px | 14px |
| `--ng-border-glow-opacity` | 0.4 | 0.6 | 0.85 |
| `--ng-pulse-scale` | 1.02 | 1.04 | 1.06 |

**New Overdrive values (deliberately non-linear push past the existing progression):**

| Field | overdrive | Reasoning |
|---|---|---|
| `--ng-glow-spread` | 20px | 1.67× intense — aggressive halo presence |
| `--ng-glow-blur` | 56px | 2× intense — dramatically extended bloom, "light pushing through the page" feel |
| `--ng-glow-opacity` | 0.95 | Nearly saturated but not 1.0 (which would flatten to solid color rather than glow) |
| `--ng-glow-text-blur` | 28px | 2× intense — text characters look like light sources |
| `--ng-border-glow-opacity` | 1.0 | Maximum |
| `--ng-pulse-scale` | 1.10 | Larger pulse amplitude |

**Why non-linear extrapolation:** Linear extrapolation from the existing pattern would give `spread 16, blur 40, opacity 0.9, text-blur 20, pulse 1.08`. That would be "extra intense" — same character, more of it. Overdrive should be qualitatively different: bloom that escapes the element silhouette, halos that interact with the page background, the "light source pushing through" effect. The non-linear push (especially blur at 2× and spread at 1.67×) creates that qualitative shift rather than just a quantitative one.

**Class block (target final form for `tokens.css`):**

```css
/* ============================================================
   INTENSITY: OVERDRIVE
   Light pushed past normal operating range. Bloom escapes the
   element boundary. The streetlamp-at-night register — what
   neon does when current exceeds spec. The brand register for
   palettes whose physical referent is itself a light source.
   ============================================================ */
.neon-overdrive {
  --ng-glow-spread:    20px;
  --ng-glow-blur:      56px;
  --ng-glow-opacity:   0.95;
  --ng-glow-text-blur: 28px;
  --ng-border-glow-opacity: 1.0;
  --ng-pulse-scale:    1.10;
}
```

### Per-theme default intensities

| Theme | Default | Reasoning |
|---|---|---|
| Rainbow | medium | Full-spectrum color is already loud; intensity shouldn't compound |
| Unicorn | intense | Dreamy fantasy register; pastels at intense produce ethereal rather than washed |
| Cinematic | intense | Blade Runner doesn't apologize |
| Pink | medium | Magenta is already assertive; medium keeps it feminine-assertive rather than aggressive |
| Retrowave | intense | 80s synthwave was designed loud |
| Grunge | intense | Raw feedback aesthetic; dial it up |
| Y2K | medium | Boy-band glossy-clean; smearing breaks the polish |
| Social | subtle | Modern corporate UI; calmness is the aesthetic |
| Cyberpunk | overdrive | Toxic dystopian neon; system saturation is the point |
| VFD | medium | Instrument display readability |
| Cherenkov | overdrive | Reactor-pool bloom needs the extra spread |
| Nixie | intense | Modern nostalgic-recreation register |

### Auto-apply mechanism

The Stimulus theme switcher controller maintains a constant mapping palette → default intensity. On palette change, the intensity is automatically set to that palette's default and the intensity dropdown updates to reflect it.

**User override semantics:** After auto-apply, the user can manually change intensity to anything they want. The next palette change will apply the new palette's default (overriding the manual choice). This means manual overrides are per-palette-session and don't persist through palette changes. This is intentional — each palette has a "right" intensity for first impression, and the user adjusting from there is welcome but doesn't override the next palette's intent.

**First-load semantics:** On first page load, if the user has a saved intensity in localStorage, that's respected (returning visitors see their last preference). If there's no saved intensity, the current palette's default is applied. This honors persistence for returning visitors while still landing intent for new ones.

**Implementation sketch (final form will be in the implementation plan):**

```javascript
const PALETTE_DEFAULT_INTENSITIES = {
  "neon-rainbow":    "neon-medium",
  "neon-unicorn":    "neon-intense",
  "neon-cinematic":  "neon-intense",
  "neon-pink":       "neon-medium",
  "neon-retrowave":  "neon-intense",
  "neon-grunge":     "neon-intense",
  "neon-y2k":        "neon-medium",
  "neon-social":     "neon-subtle",
  "neon-cyberpunk":  "neon-overdrive",
  "neon-vfd":        "neon-medium",
  "neon-cherenkov":  "neon-overdrive",
  "neon-nixie":      "neon-intense",
}

// In changePalette(): after applyPalette(), look up default intensity,
// applyIntensity() with the default, update intensity dropdown to reflect.
```

The `INTENSITIES` array in the controller also gains `"neon-overdrive"` so cleanup logic doesn't leave stale intensity classes on body when switching.

### Out of scope

- Any modification of existing intensity values (subtle, medium, intense) — they stay as-is
- Any modification of existing palette colors
- Any new palette
- Typography changes (Phase 2)
- Hardware chrome (Phase 4)
- Era chrome (Phase 5)
- VFD `--ng-text-inverse` fix (deferred to Phase 4)
- A "lock intensity" UI mechanism that prevents auto-apply on palette switch (could be added later if users request it; initial assumption is auto-apply-with-manual-override is sufficient)

## Definition of done

- `.neon-overdrive` class block exists in `tokens.css` with the exact values above
- `INTENSITIES` array in `theme_switcher_controller.js` includes `"neon-overdrive"`
- `PALETTE_DEFAULT_INTENSITIES` mapping exists in the controller
- Auto-apply triggers on palette change and on first-load-without-saved-intensity
- Navbar intensity dropdown includes `<option value="neon-overdrive">Overdrive</option>`
- `README.md` reflects 4 intensity levels everywhere it counts intensities, including the new Per-theme defaults section
- `tokens.css` header comment updated
- System tests cover: Overdrive class application, auto-apply on palette switch, manual override after auto-apply, first-load behavior with and without saved intensity
- All existing system tests still pass (no regressions on Phase 1 work)
- TK has visually verified Overdrive on at least three palettes (Cyberpunk, Cherenkov, plus one of his choice) and approved
- Branch merged to main

## Success criteria

1. Selecting Cyberpunk in the dropdown automatically displays it in Overdrive — the toxic dystopian saturation lands without the user having to know about intensity controls
2. Selecting Cherenkov automatically displays it in Overdrive — the reactor-pool bloom achieves the volumetric "light through water" effect we designed for
3. Selecting Social automatically displays it in Subtle — the corporate-UI calmness lands without competing with the palette
4. Manual intensity override still works (user can crank Cinematic to Overdrive if they want to see what it looks like)
5. Returning visitors who had a saved intensity preference see it preserved on next visit
6. New visitors see each palette in its intended intensity on first switch — no learning curve required to experience the design intent
